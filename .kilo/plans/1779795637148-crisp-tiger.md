## 1. Current vs Target Architecture

```
╔═══════════════════ CURRENT STATE (BROKEN INTEGRATION) ═══════════════════╗
║                                                                          ║
║  Fetcher(6 APIs) ────→ Scoring(deterministic only) ──→ CruxScore        ║
║                                                                          ║
║  Research(web) ──→ Verification ──→ (isolated — only Lens/Report use it)║
║                                                                          ║
║  PROBLEM: Verified evidence never reaches Scoring.                       ║
║           Scoring has no LLM component at all.                           ║
║           Lens sees score + research + verification as 3 separate things.║
╚══════════════════════════════════════════════════════════════════════════╝

╔═════════════════ TARGET STATE (UNIFIED EVIDENCE PIPELINE) ══════════════╗
║                                                                          ║
║  Fetcher(6 APIs) ──────────┐                                             ║
║                             ├──→ Verification ──→ VerifiedEvidence[]     ║

║                                                                 ↓         ║

║                                          │                               ║
║                                          ↓                               ║
║                             Lens SSE → user sees score + cited reasons   ║
║                             Report agent → narrative with evidence       ║
╚══════════════════════════════════════════════════════════════════════════╝
```

## 2. Implementation Slices (matched to IMPLEMENTATION_PROTOCOL.md)

### Slice 1: Fetcher-to-Evidence Adapter
**Goal:** Convert all 6 fetcher data sources into normalized `EvidenceItem` rows so the Verification agent can evaluate fetcher data alongside web research data.

**New file:** `src/modules/crux/shared/fetcher-to-evidence.ts`

**What it does:**
- Accepts `AggregatedFetcherOutput` + `run_id` + `property_id`
- Maps each of the 6 sources to an `EvidenceItem` with:
  - `domain`: mapped from source (`cpcb_aqi` → `environment`, `google_maps` → `locality`, `mca21` → `developer`, `ecourts` → `legal`, `nhb_residex` + `cpwd` → `market`)
  - `source_kind`: `'web'` (these are web API calls)
  - `authority_tier`: `'official'` for CPCB/MCA21/eCourts (government sources), `'primary'` for NHB RESIDEX/CPWD (semi-official), `'secondary'` for Google Maps
  - `status`: always `'accepted'` (these come from our own controlled fetchers)
  - `claim_text`: natural language description of what the data says
  - `normalized_claim`: JSON of the raw data fields
  - `confidence`: derived from fetcher success + circuit breaker health
  - `claim_hash`: SHA-256 of domain + claim + source identity

**No existing files modified.** Pure addition, minimal blast radius.

**Test plan:**
- `test/crux/fetcher-to-evidence.test.ts` — verify each of the 6 sources maps correctly
- Verify failed sources are skipped (no evidence item created)
- Verify claim_text is informative enough for Verification to analyze
- Verify authority_tier assignments are correct per source type

### Slice 2: Unified Pipeline Orchestrator
**Goal:** Run Fetcher + Research in parallel, merge results into a single evidence pool, then run Verification on the combined pool.

**New file:** `src/modules/crux/orchestrator/orchestrator.service.ts`

**What it does:**
```typescript
async function runUnifiedPipeline(property: PropertyProfile): Promise<{
  fetcherOutput: AggregatedFetcherOutput
  researchResult: ResearchRunResult
  verificationResult: VerificationRunResult
}> {
  // Step 1: Run Fetcher and Research in parallel
  const [fetcherOutput, researchResult] = await Promise.all([
    fetchAllSources(property),
    runResearch({ property_id: property.id, force_refresh: true, surface: 'api' })
  ])

  // Step 2: Convert fetcher output → evidence items, persist them
  const fetcherEvidence = fetcherOutputToEvidenceItems(fetcherOutput, researchResult.run.id)
  // Insert fetcher evidence into crux_evidence_items table (same table Research uses)
  await insertFetcherEvidence(fetcherEvidence)

  // Step 3: Verification now runs on combined evidence pool
  const verificationResult = await runVerification({
    property_id: property.id,
    force_refresh: true
  })

  return { fetcherOutput, researchResult, verificationResult }
}
```

**Key design decisions:**
- Reuses `researchResult.run.id` as the evidence run ID so fetcher evidence is linked to the same run that Verification expects
- Fetcher evidence is `status: 'accepted'` and `authority_tier: 'official'/'primary'` — Verification will naturally confirm these
- Both fetcher and research can fail independently; the pipeline continues with whatever data succeeded
- Fetcher evidence uses the `crux_evidence_items` table (same table Research writes to) — no new DB migration needed

**Test plan:**
- `test/crux/orchestrator.test.ts` — mock fetcher + research, verify parallel execution
- Verify fetcher evidence is inserted into DB and linked to research run
- Verify verification runs on the combined pool
- Verify partial failures (fetcher succeeds but research fails, and vice versa)

### Slice 3: LLM-Bounded Weight Adjustment in Scoring Agent
**Goal:** Add an LLM component to the scoring agent that adjusts category weights by tiny, bounded amounts (±0.03 per category, ±0.05 total) with specific cited reasons from verified evidence.

**Modified file:** `src/modules/crux/agents/scoring.agent.ts`

**New functions added:**
1. `adjustWeightsWithLLM()` — non-streaming Gemini call with strict output schema
2. `computeScore()` modified to accept optional `VerifiedEvidenceItem[]`

**LLM prompt design:**

```typescript
const SYSTEM_PROMPT = `
You are the CRUX Scoring Weight Adjuster. Your ONLY job is to recommend tiny adjustments to scoring category weights based on verified evidence.

CONSTRAINTS (non-negotiable):
1. You may adjust each weight by AT MOST ±0.03 from its base value.
2. The TOTAL absolute change across all 6 categories MUST NOT exceed 0.05.
3. For EVERY adjustment, you MUST cite which verified evidence item supports it (use evidence item IDs).
4. If no evidence supports an adjustment for a category, leave its delta at 0.
5. Output ONLY valid JSON. Never output reasoning text.

Weight adjustments should reflect REAL PROPERTY-SPECIFIC information:
- Positive verified evidence about location → slightly increase location_intelligence weight
- Negative verified evidence about developer → slightly increase developer_reliability weight (more weight on a weak signal signals risk)
- Verified market pricing data → slightly adjust market_valuation weight
- Environmental concerns (AQI data, flood risks) → adjust environment-related weights
`

const USER_PROMPT = `
BASE WEIGHTS:
${JSON.stringify(baseWeights)}

VERIFIED EVIDENCE (only use these — never fabricate):
${verifiedEvidence.map(e => ({
  id: e.evidence.id,
  domain: e.evidence.domain,
  claim: e.evidence.claim_text,
  confidence: e.verification.verifier_confidence,
  status: e.verification.verification_status
}))}

INTENT: ${intent}
LIFECYCLE: ${lifecycle}
MACRO_CYCLE: ${macroCycle}

Respond with:
{
  "adjustments": [
    { "category": "location_intelligence", "delta": 0.02, "reason": "Verified AQI data shows excellent air quality (evidence: abc-123)", "evidence_ids": ["abc-123"] },
    ...
  ]
}
`
```

**Validation layer (deterministic, runs after LLM):**
```typescript
function validateAndClampAdjustments(raw, baseWeights) {
  // 1. Each delta clamped between -0.03 and +0.03
  // 2. Sum of |delta| across all 6 categories ≤ 0.05
  // 3. Every evidence_id cited must exist in the input
  // 4. Adjusted weights must sum to 1.0 (re-normalize)
  // 5. No category weight may go negative
  // 6. If LLM violates any rule, use base weights (fallback to deterministic)
}
```

**New fields added to CruxScore:**
```typescript
interface CruxScore {
  // ... existing fields ...
  weight_adjustments: Array<{
    category: ScoringCategory
    base_weight: number
    adjusted_weight: number
    delta: number
    reason: string
    evidence_ids: string[]
  }>
  verified_evidence_used: number  // count of verified evidence items that influenced scoring
}
```

**Data integrity guarantees:**
- LLM cannot change individual category scores (those stay deterministic)
- LLM cannot shift any weight by more than ±0.03
- Total weight shift capped at 0.05 (about 5% of the scoring model)
- All adjustments are persisted in `score_breakdown` as `weight_adjustments` for audit
- If LLM fails/times out, scoring falls back to purely deterministic weights
- Methodology hash includes the adjustment records

**Test plan:**
- `test/crux/scoring.llm-adjust.test.ts`:
  - Verify LLM receives verified evidence correctly
  - Verify adjustments are within bounds (±0.03 per category)
  - Verify total shift ≤ 0.05
  - Verify LLM failure falls back to deterministic
  - Verify LLM returns invalid JSON → fallback
  - Verify evidence_ids cited actually exist
  - Verify normalized weights sum to 1.0
  - Verify adjustments change the composite score (integration sanity check)

### Slice 4: Wire Pipeline End-to-End in scoring/index.ts
**Goal:** Replace the current `computeAndPersist` to use the unified pipeline.

**Modified file:** `src/modules/crux/scoring/index.ts`

**Changes:**
1. `computeAndPersist()` now:
   - Fetches property profile
   - Calls `runUnifiedPipeline()` (Slice 2)
   - Calls `computeScore()` with verified evidence (Slice 3)
   - Persists score with weight_adjustments
   - Updates agent log with verified_evidence_used count

2. `getOrComputeScore()` — add `forceEvidenceRefresh` option for Lens `triggerScore` calls

```typescript
async function computeAndPersist(
  propertyId: string,
  intent: IntentProfile,
  lifecycle: LifecycleStage,
  macroCycle: MacroCycle,
): Promise<CruxScore> {
  const profile = await getPropertyProfile(propertyId)
  
  // Unified pipeline: Fetcher + Research → Verification
  const { fetcherOutput, verificationResult } = await runUnifiedPipeline(profile)
  
  // Compute score with verified evidence
  const score = await computeScore(
    fetcherOutput,
    intent,
    lifecycle,
    macroCycle,
    verificationResult.digest.verified_items  // NEW parameter
  )
  
  await persistScore(score)
  await logScoringRun(propertyId, score, verificationResult)
  
  return score
}
```

## 7. Key Design Principles

1. **Deterministic core, LLM as trusted advisor**: Individual category scores are never touched by the LLM. Only weights shift by tiny amounts, and every shift requires a cited reason.

2. **Fail open, don't fail broken**: If the LLM component fails (timeout, bad response, API error), the system falls back to pure deterministic scoring. Users get a slightly less "tuned" score but never a broken one.

3. **Auditable adjustments**: Every weight change is persisted with its supporting evidence citation. If a user questions their score, we can point to exactly which verified fact caused which adjustment.

4. **No circular dependencies**: Fetcher and Research are independent. Verification runs once on the merged pool. Scoring runs once on the verified output. Single forward pass.

5. **Fetcher data is treated as evidence**: Government API data (CPCB, MCA21, eCourts) gets `authority_tier: 'official'` and flows through the same Verification pipeline as web research. This means a government source saying "good AQI" is verified just like a news article saying "good AQI" — and if they contradict, Verification catches it.

6. **Minimal blast radius**: Only 2 new source files created, only 3 existing files modified, no database migrations required (existing JSONB columns absorb new fields).
