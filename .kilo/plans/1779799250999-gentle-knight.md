**Why this split:**
- Fetcher knows WHAT data to extract (AQI, HPI, company status, cases) and uses search/scrape/agent to pull it from specific authoritative domains
- Research uses Gemini as the brain to think creatively about WHAT questions to ask, WHERE to look, and HOW deep to go — this is the competitive advantage that competitors won't have

## 2. New Shared Infrastructure: `src/lib/firecrawl.ts`

A single Firecrawl client that both agents share. No duplication.

```
src/lib/firecrawl.ts
├── class FirecrawlClient
│   ├── search(query, options?)    → FirecrawlSearchResponse   (POST /v2/search)
│   ├── scrape(url, options?)     → FirecrawlScrapeResponse   (POST /v2/scrape)
│   ├── crawl(url, options?)      → FirecrawlCrawlResponse    (POST /v2/crawl)
│   ├── map(url, options?)        → FirecrawlMapResponse      (POST /v2/map)
│   ├── agent(prompt, options?)   → FirecrawlAgentResponse    (POST /v2/agent)
│   ├── interact(url, prompt, actions?) → FirecrawlInteractResponse (POST /v2/interact)
│   └── Private: auth headers, retry logic, timeout, circuit breaker
│
├── interface FirecrawlSearchOptions:
│   └── query: string              // search query
│   └── limit?: number             // max results
│   └── scrapeOptions?: { formats: ['markdown'] }  // auto-scrape results
│
├── interface FirecrawlScrapeOptions:
│   └── waitFor?: number           // ms to wait for JS render (SPA support)
│   └── actions?: Action[]         // click, type, scroll before capture
│   └── formats?: ['markdown']     // always request markdown
│   └── onlyMainContent?: true     // strip nav/ads/footers → high-signal data
│
├── interface FirecrawlCrawlOptions:
│   └── limit?: number             // max pages to crawl
│   └── maxDepth?: number          // link depth from start
│   └── scrapeOptions: FirecrawlScrapeOptions
│   └── includePaths?: string[]    // regex for URL patterns to include
│   └── excludePaths?: string[]    // regex for URL patterns to skip
│
├── interface FirecrawlMapOptions:
│   └── limit?: number             // max links to return
│   └── search?: string            // filter links by text match
│
├── interface FirecrawlAgentOptions:
│   └── prompt: string             // "Go to mca.gov.in, find company X..."
│   └── maxSteps?: number          // max actions agent can take
│   └── schema?: object            // structured output schema
│
└── interface FirecrawlInteractOptions:
    └── prompt: string             // "Click 'Search', type company name..."
    └── actions?: Action[]         // pre-defined click/type actions
    └── schema?: object            // structured output from the interaction
```

**Key design decisions:**
- Always use `formats: ['markdown']` — Gemini processes markdown far more accurately than HTML
- `onlyMainContent: true` for Fetcher (deterministic data extraction), `false` for Research (context can be useful)
- `/v2/agent` used for complex multi-step government portals (MCA21, RERA, eCourts) where Firecrawl's built-in browser automation handles captchas, form fills, and pagination
- `/v2/interact` used for sites requiring specific interactions (search boxes, dropdowns, paginated results)
- Built-in exponential backoff retry (3 attempts, 1s/2s/4s)
- 30-second timeout per agent/interact call (they take longer), 15s for scrape/search
- Circuit breaker (opossum) with 3 failures → 30s cooldown

### Helper function design:

```
firecrawlSearchAndExtract({ query, scrapeUrls, schema, interactPrompt?, staleFallback? }):
  1. Try /v2/search with query → get results with auto-scraped content
  2. If scrapeUrls provided, try /v2/scrape on each
  3. If interactPrompt provided, try /v2/interact on the first scrapeUrl
  4. Pass ALL collected markdown to Gemini with strict JSON schema
  5. Gemini extracts structured data from the combined content pool
  6. If Gemini returns valid data → FetcherResult { success: true, data }
  7. If ALL methods fail → staleFallback if provided, else FetcherResult { success: false }
  8. Circuit breaker wraps the entire pipeline

firecrawlAgentExtract({ prompt, schema, fallbackUrls? }):
  1. Call /v2/agent with prompt + schema → agent navigates and extracts
  2. If agent succeeds → FetcherResult { success: true, data }
  3. If agent fails, try /v2/search on fallbackUrls for the company
  4. If all fail → FetcherResult { success: false, error }
```

### Stale data fallback policy:
- **Only used when Firecrawl is completely unreachable** for a specific source
- **Always logged**: `console.warn('[fetcher] STALE_NHB_DATA: using mock HPI for {city}')`
- **Confidence degraded**: Sources using stale data contribute 0 to `sources_succeeded` count
- **NHB RESIDEX stale fallback**: Existing `RESIDEX_MOCK` data (12 cities, last updated Q3 2025)
- **CPWD stale fallback**: Existing `CPWD_RATES_MOCK` data (8 states)
- **CPCB AQI stale fallback**: Static city-average AQI table (approximate values)
- **MCA21/eCourts**: NO stale fallback — these are identity-dependent, stale data is misleading

### Phase 2: Multi-Strategy Execution

For EACH query, the research agent uses multiple strategies in parallel:

```
Strategy A: /v2/search (primary)
  → Firecrawl's native search crawls the web
  → Auto-scrapes top results for full content
  → Returns SearchResult[] with markdown content

Strategy B: /v2/map → /v2/scrape (deep domain exploration)
  → For queries with domain_hint (RERA, government portals)
  → Map the domain to discover all sub-pages
  → Filter for project/developer-specific URLs
  → Scrape each relevant page individually

Strategy C: /v2/agent (multi-step investigation)
  → For complex queries: "Find all RERA-registered projects by Shivalik Group
    across all state RERA portals and extract delivery status for each"
  → Agent navigates multiple portals, searches, extracts data
  → Returns consolidated structured evidence

Strategy D: /v2/interact (form-filling portals)
  → For portals requiring search inputs (court databases, RERA search)
  → Interact types the query, navigates results, extracts data
```

### Modified `research.web.ts` → FirecrawlWebProvider

```
class FirecrawlWebProvider implements ResearchWebProvider {
  
  async search(query: string, options?): Promise<SearchResult[]> {
    // Direct /v2/search — native Firecrawl search is now available
    const result = await firecrawl.search(query, {
      limit: options?.maxResults ?? 8,
      scrapeOptions: {
        formats: ['markdown'],
        onlyMainContent: false  // Context matters for research
      }
    })
    
    return result.data.map(r => ({
      title: r.title ?? 'Untitled',
      url: r.url ?? '',
      snippet: r.description ?? '',
      raw_content: r.markdown ?? null,
      score: null,
      published_at: null
    }))
  }
  
  async fetchPage(url: string): Promise<PageContent> {
    const result = await firecrawl.scrape(url, {
      formats: ['markdown'],
      onlyMainContent: true
    })
    return {
      url,
      title: result.metadata?.title ?? url,
      text_content: result.markdown ?? '',
      excerpt: result.markdown?.slice(0, 280) ?? '',
      fetched_at: new Date().toISOString()
    }
  }
  
  async mapDomain(url: string, search?: string): Promise<string[]> {
    const result = await firecrawl.map(url, {
      limit: 100,
      search   // filter by keyword on the domain
    })
    return (result.links ?? []).slice(0, 30)  // Top 30 most relevant
  }
  
  async deepResearch(query: string, domainHint?: string): Promise<SearchResult[]> {
    // Combine v2/search + v2/agent for deep investigation
    if (domainHint?.includes('rera') || domainHint?.includes('court')) {
      // Government portals → use /v2/agent for multi-step extraction
      const agentResult = await firecrawl.agent({
        prompt: `Navigate to ${domainHint} and search for: ${query}. 
                 Extract all relevant structured data.`,
        maxSteps: 6
      })
      // Also run regular search in parallel
      const searchResult = await this.search(query)
      return [...this.parseAgentResults(agentResult), ...searchResult]
    }
    return this.search(query)
  }
  
  async interactWithPortal(url: string, searchTerm: string): Promise<string> {
    // For portals requiring form interaction
    const result = await firecrawl.interact(url, {
      prompt: `Find the search box, type "${searchTerm}", click search, 
               wait for results, and extract all relevant data in markdown format.`
    })
    return result.markdown ?? ''
  }
}
```

### Research Flow — Complete Pipeline (Both Phases Always Run)

```
runResearch(property):
  
  PHASE 1: Gemini Query Generation (THE BRAIN)
  ────────────────────────────────────────────
  geminiQueries = gemini.generateResearchQueries({
    property: fullPropertyProfile,
    developer: developerName,
    city, state, propertyType,
    intent: scoringIntent
  })
  // → Returns 8-12 creative, targeted search queries with rationales
  
  PHASE 2: Multi-Strategy Execution (ALL in parallel)
  ───────────────────────────────────────────────────
  for each query in geminiQueries:
    
    parallelTasks = [
      // Strategy A: Direct search (always)
      firecrawlProvider.search(query.query),
      
      // Strategy B: Domain-specific deep dive (if domain_hint provided)
      query.domain_hint
        ? firecrawlProvider.mapDomain(extractDomain(query.domain_hint), query.query)
        : Promise.resolve([]),
      
      // Strategy C: Agent-based investigation (for official/legal queries)
      query.authority_tier === 'official'
        ? firecrawlProvider.deepResearch(query.query, query.domain_hint)
        : Promise.resolve([]),
    ]
    
    results = await Promise.allSettled(parallelTasks)
    
    // Collect ALL pages from ALL strategies
    for each pageUrl from all results:
      pageContent = firecrawlProvider.fetchPage(pageUrl)
      evidenceDrafts = geminiExtractor.extractEvidence({
        property, pageContent, sourceUrl: pageUrl,
        authorityTier: query.authority_tier
      })
      allEvidence.push(...evidenceDrafts)
  
  PHASE 3: Quality Filtering & Persistence (existing flow)
  ────────────────────────────────────────────────────────
  - Compute evidence status (accepted/weak/rejected) via existing policy
  - Deduplicate by claim_hash
  - Cap at CRUX_RESEARCH_MAX_EVIDENCE_ITEMS
  - Persist evidence items
  - Complete research run
  - Return ResearchRunResult with digest
```

## 6. Data Credibility — Evidence-Based Authority, Not Artificial Tiers

**The old problem:** Research evidence was always classified as `authority_tier: 'secondary'` because it came from web search. Verification then refused to mark it as `verified` because `canVerify` required `official` or `primary`. This meant research evidence could NEVER influence the score — defeating the entire purpose.

**The fix — Credibility is based on the SOURCE, not the method of discovery:**

### Dynamic Authority Classification

Research evidence is now classified based on WHERE the data was found:

```
authority_tier determination:
  
  'official':
    - Any .gov.in or .nic.in domain
    - RERA state portals (gujrera.gujarat.gov.in, maharera.maharashtra.gov.in, etc.)
    - Court databases (ecourts.gov.in, ncdrc.nic.in, consumerhelpline.gov.in)
    - CPCB, NHB, SEBI, RBI domains
    - Municipal corporation portals
    - Data sourced from Firecrawl /v2/agent on a government portal
  
  'primary':
    - Major national news outlets (Times of India, Economic Times, The Hindu, 
      Indian Express, Business Standard, Mint, Financial Express)
    - Established real estate research firms (Knight Frank, JLL, CBRE, Anarock, 
      PropEquity, Liases Foras, Cushman & Wakefield)
    - Credit rating agencies (CRISIL, ICRA, CARE, India Ratings)
    - Public company filings (BSE, NSE, Screener, ZaubaCorp, Tofler)
    - University/research institution publications
  
  'secondary':
    - Local news outlets, blogs, forums, social media
    - Unknown domains without verifiable credentials
    - User-generated content without institutional backing
```

### Verification Now Accepts ALL Tiers

**Modified `verification.service.ts:247`:**
```typescript
// OLD (broken):
const canVerify = evidence.status === 'accepted' && 
  (evidence.authority_tier === 'official' || evidence.authority_tier === 'primary');

// NEW (correct):
const canVerify = evidence.status === 'accepted';
// ANY accepted evidence can be verified — the verification quality depends on 
// the actual evidence content, not an arbitrary tier label
```

**Modified verification thresholds:**
```typescript
// official/primary sources: lower bar (inherently more trustworthy)
if (canVerify && authorityTier in ['official','primary'] && 
    direct_match && support_score >= 0.72 && contradiction_score < 0.45) {
  return 'verified';
}

// secondary sources: higher bar (need more corroboration)
if (canVerify && authorityTier === 'secondary' && 
    direct_match && support_score >= 0.82 && contradiction_score < 0.30) {
  return 'verified';
}
```

This means:
- Government data (CPCB, MCA21, eCourts) → verified with standard thresholds
- Major news/research firm data → verified with standard thresholds
- Blog/forum/social media data → verified ONLY with very high confidence AND corroboration

**Effect on scoring:**
- Research evidence from RERA portals, court databases, and government sites is now `official` tier → directly verified → flows to LLM weight adjustments
- Research evidence from credible news/research is `primary` tier → verified with standard checks
- Random web content is `secondary` tier → verified only with strong corroboration
- Net result: High-quality research evidence ACTUALLY impacts the score

### Layer 1: Firecrawl strips the noise

```
onlyMainContent: true  →  Removes <nav>, <footer>, <aside>, <script>, <style>
                           Gemini only sees the article/main content
formats: ['markdown']  →  HTML tables → clean markdown tables
                           Images → alt text, Links → [text](url)
waitFor: 3000          →  Wait for React/Vue/Angular to render before capture
```

### Layer 2: Gemini extraction with strict schemas

For FETCHER (deterministic):
- Response format: Strict JSON only, no markdown wrapping
- Schema enforcement: exact field names and types
- Invalid output → `null`, source marked as failed
- Never fabricate: "If you cannot find the data, return null for that field"

For RESEARCH (non-deterministic):
- Each claim must cite source URL + excerpt
- Confidence score (0-1) based on source authority and claim specificity
- The existing `GeminiResearchExtractor` handles this with enhanced domain classification

### Layer 3: Verification cross-checks everything

- Both fetcher-derived AND research-derived evidence verified together
- Deterministic checks: direct_match, freshness, peer conflict
- Gemini verifier: cross-references claims against property facts
- Conservative thresholds: support ≥0.72 for official/primary, ≥0.82 for secondary

### Layer 4: Scoring LLM only gets verified evidence

- `computeScore(verifiedEvidence)` — only items with `verification_status: 'verified'`
- LLM adjusts weights by ±0.03 per category, ±0.05 total
- Every adjustment cites specific verified evidence IDs → fully auditable

## 7. Resilience — Fail Gracefully, Never Break

### Pattern 1: Per-call retry with exponential backoff

```
Attempt 1: Firecrawl call → fail
  → Wait 1s
Attempt 2: Firecrawl call → fail
  → Wait 2s
Attempt 3: Firecrawl call → fail
  → Circuit breaker opens (3 failures → 30s cooldown)
  → Return error result
```

### Pattern 2: Per-source multi-strategy fallback

Each fetcher source tries MULTIPLE Firecrawl endpoints before giving up:

```
CPCB AQI:
  1. /v2/search on "{city} AQI air quality"   → if succeed: use data
  2. /v2/scrape on aqi.in/dashboard/{city}     → if succeed: use data
  3. /v2/scrape on app.cpcbccr.com             → if succeed: use data
  4. ALL failed → use stale city-average AQI table
  5. Mark source: success: false, error: CPCB_ALL_SOURCES_FAILED

MCA21:
  1. /v2/agent on mca.gov.in                    → if succeed: use data
  2. /v2/search on zaubacorp.com/{company}      → if succeed: use data
  3. /v2/search on tofler.in/{company}          → if succeed: use data
  4. ALL failed → return success: false, error: MCA21_ALL_SOURCES_UNAVAILABLE
  5. NO stale fallback (company-specific data cannot be faked)

Research queries:
  Each query tries 3 strategies in parallel:
  1. /v2/search (direct web search)
  2. /v2/map → filter → /v2/scrape (domain exploration)
  3. /v2/agent (complex investigation, if applicable)
  → Even if 2/3 fail, 1 strategy succeeding gives evidence
```

### Pattern 3: Parallel execution with Promise.allSettled

All 6 fetcher sources + all 8-12 research queries run in parallel. Individual failures don't block others. The pipeline continues with whatever data succeeded.

### Pattern 4: Stale data fallback (ONLY on total source failure)

```
Stale data is a LAST RESORT, not a default:

  NHB RESIDEX: Only if /v2/search + /v2/scrape + /v2/interact ALL fail
    → Use existing RESIDEX_MOCK (12 cities, Q3 2025 data)
    → Console.warn: "STALE_NHB_DATA: using mock HPI for {city}"
    → sources_succeeded does NOT count this as success

  CPWD: Only if /v2/search + /v2/crawl ALL fail
    → Use existing CPWD_RATES_MOCK (8 states)
    → Console.warn: "STALE_CPWD_DATA: using mock rates for {state}"

  CPCB AQI: Only if /v2/search + /v2/scrape ALL fail
    → Use static city-average AQI table
    → Console.warn: "STALE_AQI_DATA: using average AQI for {city}"

  MCA21/eCourts: NO stale fallback
    → These are identity-specific. Fake data is misleading.

  Research evidence: NO stale fallback
    → Research is about finding NEW intelligence. Stale evidence is worthless.
```

### Pattern 5: Firecrawl VM outage handling

If the Azure VM is entirely unreachable:
- Circuit breakers open for all sources after 3 failures
- Google Maps API still works independently (separate provider)
- Stale data fallbacks engage for NHB, CPWD, CPCB
- MCA21, eCourts return failure
- Research returns empty (no evidence found)
- Score computed with available data + appropriate confidence degradation
- `console.error('[firecrawl] Azure VM unreachable — pipeline operating in degraded mode')`

## 10. Key Design Principles

1. **Firecrawl is the data gateway** — Every external data point flows through Firecrawl → markdown → Gemini. Only Google Maps API is exempt (location data can't be scraped).

2. **Both agents ALWAYS run — no exceptions** — Fetcher provides the structured data foundation. Research provides the hidden intelligence that creates competitive moat. When Research fails, the pipeline continues with fetcher-only data, but the goal is for Research to succeed every time.

3. **Gemini is the brain, not just an extractor** — Gemini decides WHAT to search for, WHICH Firecrawl endpoint to use, HOW to combine results, and WHAT hidden intelligence matters. It thinks like a journalist, lawyer, and analyst simultaneously.

4. **Credibility is based on the SOURCE, not the method of discovery** — A RERA document found via web search is just as credible as one found via a hardcoded API. The `authority_tier` reflects the actual source's authority, not how we found it.

5. **ZERO mock data as primary source** — NHB RESIDEX, CPWD, and CPCB AQI must come from live Firecrawl extraction. Stale mock data is a LAST RESORT fallback, clearly logged as degraded, and does NOT count as a successful data source.

6. **Firecrawl Agent handles what APIs can't** — Government portals with captchas, search forms, and pagination are handled by `/v2/agent` (multi-step browser automation). This turns previously blocked sources (MCA21, eCourts, RERA) into reliable data streams.

7. **Every data point is traceable** — Every FetcherResult includes `data_source` URL. Every evidence item includes `source_url` + `excerpt`. Every weight adjustment cites `evidence_ids`. Full audit trail from web page to final score.

8. **Fail individual sources, never the pipeline** — `Promise.allSettled` on all 6 fetcher sources + all research queries. Circuit breakers, retries, and stale fallbacks ensure the user always gets a score.
